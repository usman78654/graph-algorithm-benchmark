3.1 GRAPH TRAVERSAL ALGORITHMS
==============================

3.1.1 Breadth-First Search (BFS)
Time Complexity:
├─ Best Case:     O(V + E)
├─ Average Case:  O(V + E)
└─ Worst Case:    O(V + E)

Space Complexity: O(V)
Implementation: collections.deque (efficient O(1) popleft)
Data Structure: Queue (FIFO)

Key Features:
- Finds shortest paths in unweighted graphs
- Explores level-by-level from source
- Returns 18,827 reachable nodes from 'leagueoflegends'
- Execution Time: 0.056346 seconds

3.1.2 Depth-First Search (DFS)
Time Complexity:
├─ Best Case:     O(V + E)
├─ Average Case:  O(V + E)
└─ Worst Case:    O(V + E)

Space Complexity: O(V) (recursion stack)
Implementation: Python list (LIFO using pop())
Data Structure: Stack

Key Features:
- Explores deeply before backtracking
- Used for cycle detection
- Returns 18,827 reachable nodes from 'leagueoflegends'
- Execution Time: 0.071730 seconds
- Performance: 1.27x slower than BFS (Python list pop(0) vs deque)

3.1.3 Cycle Detection
Time Complexity:
├─ Best Case:     O(V + E)
├─ Average Case:  O(V + E)
└─ Worst Case:    O(V + E)

Space Complexity: O(V)
Implementation: DFS-based with recursion stack tracking
Algorithm: Detects back edges using visited/recursion stack

Key Features:
- Uses color-marking (white, gray, black)
- Identifies cycle nodes
- Extremely fast: 0.000061 seconds
- Result: Cycle detected in graph

3.2 SHORTEST PATH ALGORITHMS
=============================

3.2.1 Dijkstra's Algorithm
Time Complexity:
├─ Best Case:     O((V + E) log V)  [with binary heap]
├─ Average Case:  O((V + E) log V)
└─ Worst Case:    O((V + E) log V)

Space Complexity: O(V)
Implementation: Min-heap priority queue (heapq module)
Data Structure: Priority Queue (heap)

Key Features:
- Works for non-negative weights
- Greedy approach selecting minimum distance node
- Single-source shortest path
- Execution Time: 0.129538 seconds
- Paths Generated: 18,827 destinations
- Speedup vs Bellman-Ford: 3.76x faster

Optimizations:
- Early termination when all distances finalized
- Heap operations reduce comparisons
- Efficient Python heapq implementation

3.2.2 Bellman-Ford Algorithm
Time Complexity:
├─ Best Case:     O(V + E)  [no updates needed]
├─ Average Case:  O(V * E)
└─ Worst Case:    O(V * E)

Space Complexity: O(V)
Implementation: Iterative relaxation with V-1 iterations
Data Structure: Hash tables for distance/predecessor tracking

Key Features:
- Handles negative edge weights
- Detects negative cycles
- All-to-one shortest path
- Execution Time: 0.487246 seconds
- Paths Generated: 18,827 destinations

Note: Without priority queue, both algorithms resort to brute force
checking all edges per iteration, explaining Bellman-Ford slowness.

Comparison with Dijkstra:
─────────────────────────
               Dijkstra    Bellman-Ford
Time           0.129s      0.487s
Speedup        1.0x        3.76x slower
Optimal for    Non-negative  Negative weights
Structure      Priority queue Direct iteration

3.3 MINIMUM SPANNING TREE ALGORITHMS
=====================================

3.3.1 Prim's Algorithm
Time Complexity:
├─ Best Case:     O(V log E)  [with min-heap]
├─ Average Case:  O(V log E)
└─ Worst Case:    O(V log E)

Space Complexity: O(V + E)
Implementation: Min-heap priority queue with adjacency list
Data Structure: Priority Queue (heap) + adjacency list

Key Features:
- Grows tree from starting vertex
- Greedy selection of minimum weight edge
- Connected component only
- MST Edges: 34,670
- Execution Time: 0.798253 seconds
- Total MST Weight: Sum of selected edges

Algorithm Process:
1. Start with arbitrary node
2. Add minimum weight edge to unvisited node
3. Repeat until all reachable nodes included
4. Complexity dominated by heap operations

3.3.2 Kruskal's Algorithm
Time Complexity:
├─ Best Case:     O(E log E)  [sorting dominates]
├─ Average Case:  O(E log E)
└─ Worst Case:    O(E log E)

Space Complexity: O(V + E)
Implementation: Edge sorting + Union-Find disjoint set
Data Structure: Sorted edge list + Union-Find structure

Key Features:
- Sorts all edges by weight
- Greedy selection avoiding cycles
- All components handled automatically
- MST Edges: 35,279
- Execution Time: 0.419826 seconds
- Speedup vs Prim: 1.90x faster

Union-Find Implementation:
- Path compression: O(α(n)) ≈ O(1) amortized
- Union by rank for efficient merging
- Critical for performance

Algorithm Process:
1. Sort edges by weight O(E log E)
2. For each edge in sorted order:
   - If connects different components: add to MST
   - Use Union-Find to track components
3. Early termination when V-1 edges selected

3.3.3 Average Degree Calculation
Time Complexity:
├─ Best Case:     O(1)
├─ Average Case:  O(1)
└─ Worst Case:    O(1)

Space Complexity: O(1)
Formula: Average Degree = (2 × Number of Edges) / Number of Nodes
Result: 15.7217

Execution Time: 0.000008 seconds (essentially instantaneous)

3.4 GRAPH PROPERTY CALCULATION

3.4.1 Diameter Calculation
Time Complexity:
├─ Full Graph:    O(V³)  [Floyd-Warshall approach]
├─ Limited:       O(k × (V+E)logV) where k = number of nodes
└─ Practical:     Requires optimization for large graphs

Implementation: Multi-source Dijkstra with memoization
Data Structure: Dictionary-based distances, memoization cache

Algorithm Process:
1. Run Dijkstra from k source nodes (limited to 500)
2. For each source, find maximum distance to reachable nodes
3. Track global maximum distance
4. Apply memoization to avoid redundant calculations

Results:

├─ Graph Diameter:     12.0
├─ Execution Time:     47.057217 seconds
├─ Extrapolated Full:  ~3400 seconds (all 35,776 nodes)
└─ Bottleneck:         O(V³) complexity for complete solution

Optimization Potential:
- Use Eccentricity algorithm (BFS from samples)
- Parallel processing of source nodes
- Early termination if bounds established
- Binary search on diameter value

4.3 PERFORMANCE COMPARISONS
============================

Relative Algorithm Speed:
─────────────────────────
1. DFS is 1.27x SLOWER than BFS

   - Implementation choice matters significantly

   - Bellman: O(V*E) = 35,776 × 281,229 × iterations
   - Dijkstra: O((V+E)logV) with efficient heap
   - Dijkstra optimal for non-negative weights

3. Prim is 1.90x SLOWER than Kruskal
   - Kruskal: E log E dominated by sorting
   - Prim: V log E dominated by heap operations
   - Kruskal better for dense graphs
   - Prim better for sparse graphs (but here they're similar)

5.1 QUEUE vs STACK DISCUSSION
==============================

BFS Implementation:
───────────────────
Using collections.deque:
├─ popleft(): O(1) amortized
├─ append(): O(1) amortized
└─ Optimal choice for FIFO operations
Result: BFS time = 0.056346 seconds

Alternative with list:
├─ pop(0): O(n) requires full shift
├─ append(): O(1)
└─ Poor choice for queue operations
Result would be: ~0.5+ seconds (9x slower)

CRITICAL FINDING: Queue implementation is essential!
Data Structure choice impacts BFS performance by order of magnitude.

DFS Implementation:
───────────────────
Using Python list:
├─ pop(): O(1) efficient
├─ append(): O(1) efficient
└─ Good LIFO operations
Result: DFS time = 0.071730 seconds

Performance Impact:
- Stack is well-suited for DFS
- Python list.pop() is optimal for stack
- DFS 1.27x slower than BFS due to recursion overhead

5.2 PRIORITY QUEUE IMPACT
==========================

Dijkstra with min-heap:
───────────────────────
├─ heappush(): O(log n)
├─ heappop(): O(log n)
├─ Total: O((V+E) log V)
└─ Execution: 0.129538 seconds ✓ OPTIMAL

Without priority queue (naive array):
├─ Find minimum: O(V)
├─ Per iteration: O(V²)
├─ Total: O(V²)
└─ Projected: ~100+ seconds ✗ VERY SLOW

Speedup Factor: ~775x faster with heap!
Heap is ESSENTIAL for Dijkstra efficiency.

Prim with min-heap:
───────────────────
├─ heappush(): O(log E)
├─ heappop(): O(log E)
├─ Total: O(V log E)
└─ Execution: 0.798253 seconds

Impact: Without heap would be O(V²) ≈ 100+ seconds
Priority queue selection matters critically.

5.3 UNION-FIND IMPACT ON KRUSKAL
================================

With Union-Find (path compression + union by rank):
────────────────────────────────────────────────────
├─ find(): O(α(n)) ≈ O(1) amortized
├─ union(): O(α(n)) ≈ O(1) amortized
├─ Sorting: O(E log E)
└─ Total: O(E log E) = 0.419826 seconds ✓ OPTIMAL

Without Union-Find (naive connected components):
────────────────────────────────────────────────
├─ Cycle check: O(V)
├─ Per edge: O(V)
├─ Total: O(E × V) ≈ O(V × E)
└─ Projected: ~10+ seconds ✗ SLOW

Speedup Factor: ~25x faster with Union-Find!
Union-Find is CRITICAL for Kruskal efficiency.

Path Compression Benefits:
- Flattens tree structure after find()
- Subsequent operations faster
- Nearly optimal practical performance
- Essential for large graphs

5.4 SUMMARY: DATA STRUCTURE IMPACT
===================================

Critical Data Structure Choices:
────────────────────────────────

1. Queue (collections.deque) for BFS
   ├─ Impact: 9x performance difference
   ├─ Implementation: MUST use deque
   └─ O(1) popleft essential

2. Stack (list) for DFS
   ├─ Impact: Already optimal
   ├─ Implementation: Python list good
   └─ pop() already O(1)

3. Min-Heap for Dijkstra/Prim
   ├─ Impact: 775x-100x performance improvement
   ├─ Implementation: heapq module
   └─ O(log n) operations essential

4. Union-Find for Kruskal
   ├─ Impact: 25x performance improvement
   ├─ Implementation: Path compression + union by rank
   └─ α(n) ≈ O(1) operations essential

5. Adjacency List for Graph Storage
   ├─ Impact: O(V+E) iteration vs O(V²)
   ├─ Implementation: Dict of lists
   └─ Space efficient and time efficient

7.4 ALGORITHM SELECTION GUIDE
==============================

Choose BFS when:
├─ Need unweighted shortest path
├─ Want to find all reachable nodes
└─ Performance is critical (fastest available)

Choose DFS when:
├─ Detecting cycles
├─ Topological sorting
└─ Exploring deep paths

Choose Dijkstra when:
├─ Shortest path with non-negative weights
├─ Single-source shortest paths needed
└─ Performance matters (3.76x faster than Bellman-Ford)

Choose Bellman-Ford when:
├─ Negative edge weights present
├─ Need to detect negative cycles
└─ Only option when weights can be negative

Choose Kruskal when:
├─ Need minimum spanning tree
├─ Graph is dense (better than Prim)
├─ Edge weights must be sorted anyway
└─ Prefer simple implementation

Choose Prim when:
├─ Graph is sparse
├─ Need to grow MST from specific vertex
└─ Simpler data structure preferred
